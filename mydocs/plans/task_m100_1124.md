# 수행 계획서: Task M100-1124

- 이슈: [#1124](https://github.com/edwardkim/rhwp/issues/1124)
- 브랜치: `local/task1124-hwpx-column-line`
- 대상 파일: `samples/3-11월_실전_통합_2022.hwpx`
- 비교 파일: `samples/3-11월_실전_통합_2022.hwp`
- 대상 화면: page 4 (0-index page 3), 다단 세로 구분선

## 1. 증상

작업지시자 제공 캡처 기준으로 HWPX 렌더에서 좌우 단 사이 세로 구분선이 보이지 않는다. 같은 문서의 HWP 렌더에서는 단 사이 세로선이 표시된다.

## 2. 초기 재현

소스 수정 없이 다음 명령으로 page 4 SVG를 생성했다.

```bash
./target/debug/rhwp export-svg samples/3-11월_실전_통합_2022.hwp -p 3 --debug-overlay -o output/task1124_hwp_probe
./target/debug/rhwp export-svg samples/3-11월_실전_통합_2022.hwpx -p 3 --debug-overlay -o output/task1124_hwpx_probe
./target/debug/rhwp dump-pages samples/3-11월_실전_통합_2022.hwp -p 3
./target/debug/rhwp dump-pages samples/3-11월_실전_통합_2022.hwpx -p 3
```

확인 결과:

- HWP/HWPX 모두 page 4의 본문 영역, 단 수, 단별 문단 배치는 동일하게 잡힌다.
- HWP SVG에는 `x=396.853...` 위치의 세로 단 구분선 `<line>`이 존재한다.
- HWPX SVG에는 같은 위치의 단 구분선 `<line>`이 없다.
- HWPX 원문에는 `<hp:colLine type="SOLID" width="0.12 mm" color="#000000"/>`가 존재한다.
- 현재 HWPX 파서의 `parse_col_pr()`는 `colPr`의 속성만 읽고 내부 `colLine`을 파싱하지 않는다.

## 3. 관련 코드

- HWP5 파서: `src/parser/body_text.rs::parse_column_def_ctrl`
  - `separator_type`, `separator_width`, `separator_color`를 채운다.
- HWPX 파서: `src/parser/hwpx/section.rs::parse_col_pr`
  - 현재 `type`, `layout`, `colCount`, `sameSz`, `sameGap`만 파싱한다.
- 렌더러: `src/renderer/layout.rs::build_column_separators`, `emit_zone_column_separators`
  - `PageLayoutInfo.separator_type > 0`이면 단 구분선을 그린다.

## 4. 단계 계획

1. Stage 1: 최소 원인 고정
   - HWPX `colPr` 내부 `colLine`이 파싱되지 않아 `ColumnDef.separator_type`이 0으로 남는 것을 테스트로 고정한다.
2. Stage 2: HWPX `colLine` 파싱
   - `parse_col_pr()`가 Empty `colPr`뿐 아니라 Start `colPr` + 내부 자식 요소를 처리할 수 있도록 조정한다.
   - `hp:colLine`의 `type`, `width`, `color`를 `ColumnDef.separator_*`에 매핑한다.
3. Stage 3: SVG 재검증
   - HWPX page 4 SVG에 단 구분선 `<line>`이 생성되는지 확인한다.
   - HWP 쪽 기존 단 구분선 출력과 좌표가 일치하는지 비교한다.
4. Stage 4: 회귀 검증
   - HWPX 파서 단위 테스트.
   - `cargo test --lib renderer::layout -- --nocapture`
   - 관련 HWPX roundtrip/adapter 테스트 중 영향 범위 테스트.
5. Stage 5: 단계 보고서와 최종 보고서 작성.

## 5. 위험 평가

| 위험 | 평가 | 대응 |
|------|------|------|
| `colPr` 처리 흐름 변경으로 section parsing 이벤트 소비가 어긋날 가능성 | 중간 | Start/Empty 케이스를 분리하고 기존 `skip_element` 호출 위치를 점검한다. |
| HWPX 선 굵기 문자열(`0.12 mm`) 매핑이 기존 HWP width code와 다를 가능성 | 중간 | 기존 border width 파서/유틸을 재사용하거나 최소 매핑을 테스트로 고정한다. |
| serializer roundtrip에 영향을 줄 가능성 | 낮음~중간 | HWPX 관련 테스트를 함께 실행하고, 필요 시 serializer는 별도 범위로 분리한다. |
