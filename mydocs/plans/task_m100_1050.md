### 2.3 IR 직접 확인 — Footnote model 보존됨

```
=== samples/footnote-tbox-01.hwp (한컴 정답지) ===
  sec0/p1/ctrl0 TextBox p[0]/ctrl[0]: Footnote (number=1, paragraphs=1)
    fn p[0]: cc=19, text="  글상자 내부 각주"

=== output/poc/issue_1050/footnote-tbox-01-from-hwpx.hwp (rhwp 저장본) ===
  sec0/p1/ctrl0 TextBox p[0]/ctrl[0]: Footnote (number=1, paragraphs=1)
    fn p[0]: cc=20, text="  글상자 내부 각주 "
```

→ Footnote IR (number + paragraphs) 은 정상 보존. **본질은 record level**.

### 2.4 본질 — Footnote CTRL_HEADER + LIST_HEADER payload 잘림

`rhwp hwp5-inventory-diff` + raw byte 직접 확인:

#### CTRL_HEADER (tag 71) — Footnote
| 필드 | 한컴 정답지 (size=20) | rhwp 저장본 (size=6) |
|------|---------------------|---------------------|
| `ctrl_id` (4) | `20 20 6e 66` (`fn  ` LE) | 동일 |
| `number` (2) | `01 00` | 동일 |
| **추가 14 byte** | 한컴: `00 00 00 00 29 00 00 00 00 00 34 00 00 00` (글상자 안) / `00 00 00 00 29 00 00 00 00 00 10 01 00 00` (본문) | **누락** |

#### LIST_HEADER (tag 72) — Footnote 안 paragraph 리스트
| 한컴 정답지 (size=16) | rhwp 저장본 (size=6) |
|---------------------|---------------------|
| `01 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00` (para_count=1 + 추가 12 byte) | `01 00 00 00 00 00` (잘림) |

### 2.5 근본 원인 위치 정확화

**파서 측 누락** (`src/parser/control.rs:519 parse_footnote_control`):
```rust
if ctrl_data.len() >= 2 {
    let mut r = ByteReader::new(ctrl_data);
    footnote.number = r.read_u16().unwrap_or(0);  // 2 byte만 읽고 14 byte 버림
}
```

**시리얼라이저 측 누락** (`src/serializer/control.rs:613 serialize_footnote`):
```rust
fn serialize_footnote(fn_: &Footnote, level: u16, records: &mut Vec<Record>) {
    let mut w = ByteWriter::new();
    w.write_u16(fn_.number).unwrap();  // 2 byte만 쓰고 추가 14 byte 없음
    records.push(make_ctrl_record(tags::CTRL_FOOTNOTE, level, w.as_bytes()));
    serialize_list_header_with_paragraphs(&fn_.paragraphs, level + 1, records);  // LIST_HEADER 짧은 형식
}
```

**Footnote model 누락** (`src/model/footnote.rs:8`):
```rust
pub struct Footnote {
    pub number: u16,
    pub paragraphs: Vec<Paragraph>,
    // 추가 14 byte 보존 영역 없음
}
```

LIST_HEADER 형식은 `serialize_list_header_with_paragraphs` 가 짧은 형식 (size=6) 만 출력. Footnote 의 LIST_HEADER 는 16 byte 형식 필요.

## 3. 본질 한계 공유 — Task #178 와의 관계

본 task 는 **각주 영역 한정 부분 해결** 영역:

- ✓ 한컴 열기 성공 (Task #178 광범위 거부 영역과 다름)
- ✓ 본 환경 자기 라운드트립 거의 정합 (글상자 안 각주 본문 누락만)
- ✓ IR 보존됨 (number + paragraphs)
- ✓ 본질 위치 정확 식별 (footnote CTRL_HEADER + LIST_HEADER payload 잘림)

`project_hwpx_to_hwp_adapter_limit` 메모리 룰 정합 — 광범위 한컴 호환은 어렵지만, **각주처럼 본질이 정확한 영역은 부분 정정 가능**.

## 7. 위험 / 완화

| 위험 | 완화 |
|------|------|
| 한컴 footnote payload 의 의미 미규명 시 raw 보존만으로 부족 가능 | Stage 1 에서 다른 fixture 와 비교하여 패턴 식별. 본 sample 에 한정 fix는 가능 |
| HWPX 출처 IR 에는 추가 14 byte 정보 부재 → 합리적 default 필요 | 한컴 정답지의 default 패턴 (`00 00 00 00 29 00 00 00 00 00 ...`) 추적. HWPX→HWP 매핑 contract 추가 |
| LIST_HEADER 짧은 형식이 다른 컨테이너 (cell / textbox) 와 공유될 가능성 | parser/serializer 두 형식 양립 (size 기반 분기) |
| 광범위 sweep 에서 다른 컨테이너 각주 회귀 가능 | sweep + 회귀 가드 양쪽으로 검증 |

## 8. 비범위 / 후속

- 미주 (Endnote) — 동일 본질 추정 (`serialize_endnote` 도 number 만). 필요 시 본 task 에 포함 또는 후속 분리
