# 수행계획서 — Task #986

## 이슈

**#986** 가로 방향 문서 열 경우, 표가 밑으로 밀리는 현상

- 브랜치: `issue-986-landscape-table-flow` (← 최신 `upstream/devel`)
- 마일스톤: M100 (v1.0.0)
- GitHub: https://github.com/edwardkim/rhwp/issues/986
- 제보 환경: rhwp 0.7.12, Chrome, Windows 11

## 현상

가로 방향 HWP 문서에서 여러 개의 표가 한 페이지 안에 배치되어야 하지만,
일부 표가 아래로 밀려 다음 페이지로 넘어간다.

제보 첨부 `receipt.hwp` 기준 현재 관찰:

- 문서 자체는 A4 가로 방향으로 정상 파싱된다.
- 첫 페이지 body 폭은 가로 문서 기준으로 충분하다.
- 첫 문단 안에 비글자취급 표 여러 개가 들어 있다.
- 왼쪽 큰 표와 오른쪽 작은 표들은 서로 다른 가로 영역에 배치되어야 한다.
- 현재 pagination/layout 은 이를 하나의 세로 흐름으로 취급해 오른쪽 표를
  왼쪽 표 아래로 밀어낸다.

## 원인 가설

핵심 원인은 용지 방향 파싱이 아니라 **비글자취급 `TopAndBottom` 표의
점유 영역 계산 방식**이다.

현재 구현은 `treat_as_char=false`, `wrap=TopAndBottom` 표가 나오면 표의
가로 위치와 폭을 충분히 고려하지 않고 전역 세로 커서를 밀어낸다. 이 때문에
가로로 겹치지 않는 표들까지 서로 같은 세로 흐름에 놓인 것으로 판단된다.

예상 수정 방향:

1. 표의 배치 박스를 `(x0, x1, y0, y1)` 로 계산한다.
2. 같은 문단/컬럼 안에서 가로 범위가 겹치는 표끼리만 서로 밀어낸다.
3. 가로 범위가 겹치지 않는 표는 같은 페이지의 병렬 영역에 남도록 한다.
4. pagination 과 실제 layout 이 같은 판단을 사용하게 맞춘다.

## 추가 발견

제보 첨부 파일은 debug 빌드에서 별도 panic 을 유발한다.

- 위치: `src/renderer/composer.rs`
- 증상: `text_end - text_start` overflow
- 추정 원인: 일부 표 셀 문단의 `LineSeg.text_start` 가 비단조/이상값으로
  들어와 `text_end < text_start` 가 된다.

이 panic 은 release/WebAssembly 의 화면 밀림과는 별도 문제지만, 같은 샘플의
검증 안정성을 위해 방어 로직을 함께 검토한다.

## 검증 기준

최소 성공 조건:

- 제보 첨부 `receipt.hwp` 가 가로 1페이지 구성에 맞게 렌더링된다.
- 오른쪽 작은 표가 왼쪽 큰 표 아래로 밀려 별도 페이지로 넘어가지 않는다.
- `dump-pages` 에서 본 이슈의 오른쪽 표가 불필요하게 `PartialTable` 로
  분할되지 않는다.
- debug 빌드에서도 동일 샘플 로드 중 overflow panic 이 발생하지 않는다.

회귀 확인:

- 기존 표 pagination 테스트 통과
- 기존 SVG snapshot 또는 관련 golden 테스트 영향 확인
- 가로로 겹치는 `TopAndBottom` 표는 여전히 서로 밀어낸다.

## 위험 요소

- pagination 과 layout 이 서로 다른 좌표 계산을 사용하면 페이지 수와 실제
  렌더 위치가 다시 어긋날 수 있다.
- `TopAndBottom` 의 밀어내기 의미를 너무 약하게 만들면 실제로 겹치는 표나
  본문 텍스트가 겹칠 수 있다.
- signed offset 보정은 기존 양수 offset 문서에 영향이 없어야 한다.
- 제보 첨부 파일을 저장소 fixture 로 포함할지 여부는 용량/라이선스 측면에서
  구현계획서에서 별도 확정한다.
