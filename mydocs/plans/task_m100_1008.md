## 2. 문제 정의

`samples/hwp3-sample16.hwp` 의 cover (p1) + 사업개요 (p2) 영역에서 4 격차:

### 2.1 격차 A: HWP3 Shape 박스 배경 gradient 누락 (HWP3 한정)

- 현 rhwp HWP3: `채우기: Solid` (gradient 데이터 부재)
- 한컴 정답: gradient (보라/라벤더)
- HWP5 변환본: gradient 정상 (참고)

HWP3 parser 의 Shape control fill 추출이 gradient color/positions/angle/center 데이터를 누락.

### 2.2 격차 B: Shape border style 실선/점선 (HWP3 + HWP5 공통)

- 현 rhwp: 둘 다 dashed (점선)
- 한컴 정답: solid (실선)

dump:
- HWP3: `선 style=0x0002`
- HWP5: `선 style=0xc0010043`

둘 다 점선 LineType 으로 해석되나, 한컴 동작은 실선. parser 또는 renderer 의 LineType 비트 해석.

### 2.3 격차 C: HWP3 HEAD numbering 라벨 형식 (HWP3 한정)

- 현 rhwp HWP3: "■1.추진목적■" (block markers + 형식 깨짐)
- 한컴 정답: "1. 추진목적"
- HWP5 변환본: 정상

HWP3 parser 의 HEAD type/numbering style 추출 결함.

### 2.4 격차 D: HWP3 한글 단어 공백 누락 (HWP3 한정)

- 현 rhwp HWP3: "세계3대물서비스기업"실현을 위한
- 한컴 정답: "세계 3대 물 서비스 기업" 실현을 위한
- HWP5 변환본: 정상

HWP3 parser 의 text run 추출 시 한글 단어 사이 공백 누락. text 처리 영향 광범위 가능 — Stage 1 진단 후 task 범위 유지/분리 재결정.

## 3. Root cause 가설

### 3.1 격차 A

HWP3 Shape control binary 의 fill_type=Gradient 분기에서 gradient color list / positions / angle / center 등 sub-record 가 별도 비트 stream 으로 저장. 현 HWP3 parser 가 이를 추출하지 않고 fill_type 만 읽어 Solid 로 fallback.

### 3.2 격차 B

HWP3/HWP5 border attr 의 LineType 비트 위치/값 해석이 한컴 동작과 불일치. 특히 width=56 (HU) + style=0x0002 (HWP3) 의 조합에서 한컴이 어떤 비트로 실선/점선을 결정하는지 단언 필요.

### 3.3 격차 C

HWP3 의 HEAD numbering 은 별도 record (HWPTAG_NUMBERING 또는 HWP3 전용 tag) + 라벨 형식 spec 에 저장. 현 parser 가 미인식하여 raw 텍스트 그대로 렌더.

### 3.4 격차 D

HWP3 의 한글 text 디코딩 시 공백 char(0x20) 또는 너비공백 처리 결함. 또는 char shape run 분할 시 공백을 다른 run 으로 이동 후 누락. HWP5 변환본 parser 의 동일 데이터 디코딩 코드 비교 필요.
